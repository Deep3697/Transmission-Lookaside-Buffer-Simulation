Before diving into the three modules, we must establish how the website manages data. This is what makes your simulation feel like a cohesive journey rather than three disconnected web pages.
1. The Global State (The Shared Brain)
Behind the scenes, the simulation uses a shared memory bank (like React Context). If the user generates "Page 5" in the Paging simulation, the system remembers this. When they navigate to the TLB simulation, the system will automatically use "Page 5" to test the TLB.

Phase 1: Paging Simulation (The Foundation)
The Goal: Teach the user why memory is divided into blocks and how the CPU calculates exactly where a specific piece of data lives in physical RAM.
1. Theoretical Concept & Simple Example
•	The Theory: Programs are too large to fit into continuous blocks of RAM because RAM gets fragmented over time. The OS solves this by chopping the program into equal-sized "Pages" and chopping the RAM into equal-sized "Frames."
•	Simple Example: Imagine writing a 10-page essay. Instead of needing 10 empty folders perfectly lined up in a row in your filing cabinet, you put Page 1 in folder 8, Page 2 in folder 99, and Page 3 in folder 2. As long as you keep a "Master List" (the Page Table) of where everything is, your essay is perfectly safe.
2. Visual Simulation Walkthrough
Step A: The Giant Address and the Split
•	Visual: A massive, pulsing number appears in the CPU box: Logical Address: 4196.
•	Explanation Shown: "The CPU wants data at address 4196. But RAM is chopped into blocks (Pages) of 1000 bytes. We need to split this address to find out which Page it belongs to."
•	The Animation: The user clicks "Split Address." The number 4196 physically breaks in half. The 4 turns blue and gets a label: Page Number ($p$). The 196 turns orange and gets a label: Offset ($d$).
•	The Stash: Both $p = 4$ and $d = 196$ shrink and fly into the Sidebar Variable Stash.
Step B: The Page Table Lookup
•	Visual: The blue $p = 4$ flies from the Sidebar to the "Page Table" box in the center of the screen.
•	Explanation Shown: "The OS checks the Master List (Page Table) to see where Page 4 was hidden in the physical RAM."
•	The Animation: A grid appears. Row 4 lights up brightly. It reveals that Page 4 is stored in Frame 12.
•	The Stash: A new green variable, Frame Number ($f = 12$), pops out of the table and flies into the Sidebar.
Step C: The Physical Address Calculation (The Crash)
•	Visual: The screen darkens, and a glowing math formula shell appears in the center:
$$Physical Address = (f \times Page Size) + d$$
•	Explanation Shown: "Now we calculate the exact physical location in the RAM chips."
•	The Animation: The variables $f = 12$ and $d = 196$ shoot out of the Sidebar. They slam into the formula perfectly in their respective spots. The math calculates step-by-step before the user's eyes: (12 * 1000) + 196 = 12196.
•	The Resolution: The final number 12196 flies like a missile directly into the RAM box, lighting up the exact data block the CPU requested.

Phase 2: Cache Memory Simulation (The Speed Booster)
The Goal: Step away from Paging momentarily to understand why computer engineers place a tiny, ultra-fast memory chip directly next to the CPU.
1. Theoretical Concept & Simple Example
•	The Theory: The CPU is millions of times faster than Main Memory (RAM). If the CPU has to wait for RAM every time it needs data, the whole computer crawls. A Cache is a tiny, expensive memory built into the CPU to hold the most frequently used data.
•	Simple Example: You are a chef (CPU). The Cache is a tiny spice rack right next to your cutting board. RAM is the giant basement pantry. If you need salt, grabbing it from the spice rack takes 1 second (Cache Hit). If you need saffron and it's not on the rack, you have to walk all the way to the basement, which takes 5 minutes (Cache Miss).
2. Visual Simulation Walkthrough
Step A: The Rapid Requests (Hits and Misses)
•	Visual: The screen shows a fast CPU, a small Cache box (3 empty slots), and a massive RAM box. The user is given a "Generate Request" button.
•	Explanation Shown: "Click the button to make the CPU ask for data. Watch where it goes."
•	The Animation: The user clicks. The CPU asks for "Block A".
o	Miss: A red scanner checks the Cache. It fails. A slow, red dotted line goes to RAM, grabs Block A, copies it to the Cache, and gives it to the CPU.
o	Hit: The user clicks again, asking for "Block A" a second time. A green scanner checks the Cache, finds it instantly, and a lightning-fast green line shoots back to the CPU. RAM is never touched.
Step B: The Eviction Policy (LRU)
•	Visual: The user keeps clicking until the 3 Cache slots are full (Blocks A, B, and C). The CPU asks for "Block D".
•	Explanation Shown: "The Cache is completely full! To bring in Block D from the basement, we have to throw something out. The OS uses the Least Recently Used (LRU) rule."
•	The Animation: A timer appears over Blocks A, B, and C showing how long they have been ignored. Block A has the oldest timer. A visual mechanical arm pushes Block A out of the box, and Block D slides into its place.
Step C: The Hit Ratio Calculation (The Crash)
•	Visual: A scoreboard appears showing the user's total clicks: 8 Hits, 2 Misses (Total 10 requests).
•	Explanation Shown: "How effective was our tiny spice rack? We measure this with a Hit Ratio."
•	The Animation: The formula shell appears:
$$Hit Ratio (H) = \frac{Hits}{Total Requests}$$
•	The Stash: The numbers 8 and 10 fly from the scoreboard into the formula, calculating $H = 0.8$ (or 80%). The variable $H = 0.8$, along with assumed speeds $C = 10ns$ (Cache Time) and $M = 100ns$ (Memory Time), fly into the Sidebar Variable Stash to be saved for the final boss simulation.

Phase 3: TLB Simulation (The Ultimate Merger)
The Goal: Bring Paging and Cache together. Show that checking the massive Page Table in RAM (from Phase 1) is too slow, so we build a specialized Cache (from Phase 2) just for the Page Table.
1. Theoretical Concept & Simple Example
•	The Theory: Address translation (finding the Frame) requires a trip to RAM just to read the Page Table. To avoid this penalty, the CPU uses a Translation Lookaside Buffer (TLB)—a cache that stores the most recent Page-to-Frame translations.
•	Simple Example: Think of the giant Page Table as the Master Index Catalog at the front desk of a massive library. Walking there to find every single book takes hours. The TLB is a small sticky note you keep in your pocket. If you need a Biology book, you check your sticky note first. If it's blank (TLB Miss), you walk to the front desk. But once you find out Biology is on Floor 3, you write it on your sticky note. Next time, you just read the note and go straight to Floor 3 (TLB Hit).
2. Visual Simulation Walkthrough
Step A: Reusing the Global State
•	Visual: The screen shows the CPU, a small TLB box, the Page Table in RAM, and the Physical RAM.
•	The Automation: Because the systems are connected, the Sidebar automatically slides out, revealing the $p=4$ and $d=196$ variables the user calculated back in Phase 1, along with the speeds ($H, C, M$) from Phase 2.
Step B: The TLB Miss (The Slow Path)
•	Explanation Shown: "The CPU needs Page 4. It checks the high-speed TLB sticky note first."
•	The Animation: The $p=4$ flies from the Sidebar to the TLB. The TLB grid is empty. A red "MISS" flashes.
•	The Update: The simulation takes the slow path to the giant Page Table. It finds Frame 12. Crucially, before doing anything else, it visually copies the mapping [Page 4 = Frame 12] back into the small TLB box.
Step C: The TLB Hit (The Fast Path)
•	Explanation Shown: "Because programs read code sequentially, the CPU asks for the next line of code, which is also on Page 4."
•	The Animation: The CPU sends $p=4$ to the TLB again. This time, a bright green "HIT" flashes. It grabs Frame 12 instantly and completely bypasses the giant Page Table.
Step D: The Grand Finale Calculation (Effective Access Time)
•	Visual: The screen darkens. It's time to prove mathematically why the entire operating system relies on this TLB to survive.
•	Explanation Shown: "What is the true speed of our memory system now? We calculate the Effective Access Time (EAT)."
•	The Animation: The most complex formula shell appears in the center of the screen:
$$EAT = (H \times (C + M)) + ((1 - H) \times (C + 2M))$$
•	The Ultimate Crash: Every single variable we have collected across all three simulations ($H=0.8$, $C=10$, $M=100$) shoots out of the Sidebar Stash at once. They lock into the LaTeX equation.
•	The Math: The math resolves dynamically on screen:
o	Hit Path: $0.8 \times (10 + 100) = 88ns$
o	Miss Path: $0.2 \times (10 + 200) = 42ns$
o	Result: $130ns$ average access time.
•	The Conclusion: The popup displays the final message: "Without the TLB, every request would take 200ns. By using the TLB cache, we nearly doubled the speed of the computer!"
